BOTServer

Telegram Bots Platform ?

Please read offical documentation:

  • Telegram Bot Platform Revolution!

  • Telegram Bot API

Why ?

  • First goal: set-up a Ruby way to manage Telegram Bot API 'getUpdates' via HTTPS webhooks, a performance improvement in comparison with HTTPS long polling. So let consider this as an appendix of telegram-bot-ruby gem, Alexander Tipugin's excellent Telegram Bot APIs wrapper; to thank him for his work, I tried to solve issue #19: setting up webhooks with telegram-bot-ruby ?. BOTServer here is to close the open issue :)

  • Second goal: receive https callbacks is probably more efficient than getting updates on a long polling connection, but the real need of webhooks is when you have many bots to manage at once. Let say dozen, hundreds of bots! In this scenario, it could pretty impossible to manage in a single host, so many long polling connections, because we have to maintain open too many HTTPS persistent connections for long polling mode.

More details about long polling vs webhooks in wiki document: Telegram Bot Architecture(s)

A lot of bots ?

You are maybe asking yourself, what the hell of application needs so many bots!? Let consider a sort of e-commerce marketplace, where your backend bot server must manage multiple bots, where each bot "represent" any shop, any service supplier, any merchant interface! Confused ? I reported some examples of bots usage in humans-to-humans ineternedaition, see: Innovative Chatbot Services with Telegram. By example consider a "Pizza maker in-store shop" scenario, where a buyer do an online order chatting with a virtula shop bot, that helps buyer filling order and sending order to seller! So, for this "marketplace" it make sense to manage a multitude of bots (1 bot = 1 shop) with an efficient server architecture. For this specific business reason I created BOTServer: a sort of webhooks handler microframework to serve "a lot of bots".

Nuts and bolts

BOTServer provides some command line scripts to build the game. Just run rake from your project home: 

$ rake
rake app:new[token]        # create bot app template for given token
rake certificate:new       # create certificate
rake certificate:show      # show public certificate
rake proxy:new             # show nginx proxy configuration for ssl
rake rack:log              # show rack sever logfile: /your_home/BOTServer/log/thin.log
rake rack:pid              # rack server pid
rake rack:restart          # restart rack server
rake rack:stop             # stop rack server
rake server:check          # check server configuration: /your_home/BOTServer/config/server.yml
rake server:show           # show server configuration: /your_home/BOTServer/config/server.yml
rake tokens:check          # verify tokens, online querying Telegram Server
rake tokens:show           # show tokens configuration file: /your_home/BOTServer/config/tokens.yml
rake webhook:reset[token]  # Reset webhook for given token
rake webhook:set[token]    # Set webhook for given token

In next paragraph I'll show you step by step, explaining all in detail what commands are for.

BOTServer run-time engine is very simple: it's a rack application (now implemented with Sinatra, but I'll soon substitute it with flat fast rack app), that dispatch dinamically incoming token webhooks, calling update method of an instance of a pregenerated class, that define the bot. So, yes, I used a bit Ruby on Rails metaprogramming paradigma for the game. Here below the dispatcher (rackup/sinatra.rb) code extract:

# load bots in memory. return a lookup table
set lookup: Loader.create_lookup(Config.tokens_config_file)

# HTTPS POST webhook endpoint(s)
post '/:token' do | token |

  # retrieve name, method object pair from lookup table
  bot = settings.lookup[token.to_sym]

  unless bot.nil?
    request.body.rewind
    body = request.body.read
    data = MultiJson.load(body)
    update = Telegram::Bot::Types::Update.new(data)

    # TRICKY:  object method run-time execution
    bot[:method].call(update)
  end
end

Assembly instructions in 7 steps

BOTServer is nothing else than a sequential devops tasks and a runt-time webhooks dispatcher. I list here and I'll explain in deep in next paragraps:

  1. Install a web/proxy server, Ruby project stuff
  2. Create (self-signed) Certificate
  3. Get your Telegram Bot token(s)
  4. Set-up configuration files
  5. Set "Webhooks mode" for each token
  6. Generate template for each bot
  7. Deploy and run BOTServer

Step 1. Installation

Install your preferred proxy-server

This is a pro-Soviet project (I'm joking) and we love Igor Sysoev's NGINX web server creature!:-). Jokes a part, using a front end server is not mandatory. By example my preferred rack server, Thin is able to manage HTTPS and certificates, but there is a security concern here: when receiving webhooks, you expose your own server to possible attacks, and it's better to do not expose to crazy internet your preferred Ruby rack server (Thin, Puma, whatever).

NGINX is a great fast and robust front-end firewall and load balancer. Last but not least, NGINX know how to manage SSL certificates and all digital certificates' cross-validations.

NGINX installation depends on your OS. Please refer to NGINX website for download and installation info.

See architectural overview here behind: NGINX act as a SSL manager, proxy and load balancer in front of one or many rack servers (workers).

$ git clone https://github.com/solyaris/BOTServer.git
$ cd BOTServer && bundle install

Step 2. Create (self-signed) Certificate

I prepared a rake command to just show on command line terminal an openssl command to build-up secret an publick key. Please copy/paste to generate your self-signed certificate, modifying the shown command with your personal data:

$ rake certificate:new

Step 3. Get your Telegram Bot token(s)

You have to chat with Telegram Bot Father, to create each of your bot! Afterward, take your secret tokens on your secret files.

Step 5. Set Webhooks

Set webhook for each token

$ rake webhook:set[YOUR_TOKEN_1]
$ rake webhook:set[YOUR_TOKEN_2]
$ rake webhook:set[YOUR_TOKEN_3]
$ rake webhook:set[YOUR_TOKEN_N]

BTW, if you want to reset a webhook, to go back with long polling (EXCLUSIVE) mode, for example for bot associated with token YOUR_TOKEN_N, just run command: 

$ rake webhook:reset[YOUR_TOKEN_N]

Step 6. Generate template for each of your application bots

create bot app template for each token:

$ rake app:new[YOUR_TOKEN_1]
$ rake app:new[YOUR_TOKEN_2]
$ rake app:new[YOUR_TOKEN_3]
$ rake app:new[YOUR_TOKEN_4]

My idea here, is to generate an app template, to be filled with your own logic. The generated app implement just an echo-server trivial logic. Of course you have to link-up here, in the update method your real bot LOGIC!

#
# file: yourappexample.rb 
#
require 'telegram/bot'

class Yourappexamplebot 
  attr_reader :token, :client

  def initialize
    @token = 'YOUR_APP_EXAMPLE_TOKEN'
    @client = Telegram::Bot::Client.new(@token)
  end

  #
  # message(s) updates from telegram server.
  # put ALL your Telegram Bot logic here.
  #
  def update(data)
    update_id = data.update_id
    message = data.message
    message_id = message.message_id 

    #
    # #######################
    # PUT HERE YOUR BOT LOGIC
    # #######################
    #    

    # echo-server, just for test purpose
    case message.text
    when /.+/ 
      text = "#{message.from.first_name}:#{message.text}"
      chat_id = message.chat.id

      # send echo to user 
      client.api.send_message(chat_id: chat_id, text: text)
      puts "#{update_id}:#{message_id}:#{text}"
    end  
  end


  alias get_update update
end

Step 7. Deploy and Run

  • Start web server
  • Start rack server
  • Monit incoming webhooks
$ service nginx restart
$ rake rack:restart
$ rake rack:log
